# الفصل الرابع: التنفيذ والاختبار (Chapter 4: Implementation and Testing)

## 4.1 Introduction (مقدمة)
ينتقل هذا الفصل من مرحلة التصميم النظري إلى مرحلة التنفيذ البرمجي الفعلي. يتم فيه تسليط الضوء على بيئة التطوير، والواجهات البرمجية الأساسية التي تم برمجتها، إضافة إلى عرض الواجهات الفعلية للنظام بعد اكتمالها (Screenshots). كما يناقش التحديات التقنية التي واجهتنا أثناء التطوير وكيفية حلها، ويختتم بمنهجيات اختبار النظام للتأكد من جاهزيته للعمل الميداني.

## 4.2 Development Environment (بيئة التطوير والأدوات)
لبرمجة المشروع تم تحضير البيئة التالية:
* **نظام التشغيل:** Windows.
* **برامج تحرير الأكواد:** Visual Studio Code مع إضافات متخصصة لـ Python والـ React.
* **خادم الويب وقواعد البيانات:** تم الاعتماد على خادم Django الداخلي للتطوير مع سيرفر MySQL لبيئة التجارب.
* **أدوات إدارة الـ API:** تم استخدام متصفح النظام وأدوات مثل أدوات المطورين في المتصفح لاختبار مخرجات REST APIs.

## 4.3 Implementation Details (تفاصيل التنفيذ)
تم تقسيم النظام برمجياً إلى هيكلين منفصلين يتخاطبان عبر واجهة اتصال (API):

### 4.3.1 Backend APIs (برمجة الواجهة الخلفية)
في Django، قمنا ببناء عدة تطبيقات مصغرة (Apps) لتقسيم الكود، مثل:
* تطبيق التطبيقات المستخدمين (`users`): للتحكم بعمليات التسجيل وصلاحيات الـ (Token).
* تطبيق جدولة الأوقات (`scheduling`): لإدارة أوقات عمل الطبيب الفعالة والمتاحة.
* تطبيق الحجوزات (`appointments / clinic`): لإصدار وحفظ المواعيد الفعلية في الـ MySQL.

### 4.3.2 Frontend Components (برمجة الواجهة الأمامية)
في بيئة React، تم استخدام نمط (Component-Based) مثل `SecretarySchedulePage.jsx` لإدارة شاشة السكرتارية المستقلة. وتم استهلاك البيانات بفضل مكتبة Axios التي ترسل الـ Token في عنوان الطلب (Headers) لضمان الخصوصية.

## 4.5 Implementation Challenges (التحديات التقنية التي واجهتنا)
أثناء مرحلة التنفيذ، واجهنا بعض التحديات التقنية التي تطلبت تخطيطاً هندسياً لحلها، ومن أبرزها:
* **تحدي المزامنة بين واجهة الطبيب و "حجوزاتي" وإدارة السكرتارية:**
حدثت مشكلة في البداية تتمثل في تضارب عملية تزامن البيانات عند إلغاء أو إنشاء حجز بشكل مباشر من قبل السكرتير أثناء تصفح المريض للواجهة. **طريقة الحل:** قمنا بضبط دوال الـ (States) في منصة React والمزامنة الدقيقة لطلبات الـ API لضمان أن الحجز الملغى ينعكس مباشرة في جداول العرض دون التسبب في أخطاء أو ترك حقول محجوزة وهمية.

## 4.6 System Testing (اختبار النظام)
تم إخضاع النظام لعدة اختبارات لضمان جودته:
* **Unit Testing (اختبار الوحدات):** التأكد من أن دوال مثل (منع إنشاء موعد في يوم إجازة) تعمل بشكل صحيح عبر إرسال بيانات خاطئة ورؤية رد النظام (وجوب رد النظام بخطأ 400 Bad Request).
* **Integration Testing (اختبار التكامل):** التأكد من أن عملية الحجز من الـ Frontend تصل وتُسجل بنجاح في قاعدة البيانات MySQL عبر الـ Backend.
